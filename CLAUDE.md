# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm install        # once
npm start          # dev server + hot reload at http://localhost:3000
npm run build      # static build into build/ — MUST finish with zero broken-link warnings
npm run serve      # serve the production build to spot-check
npm run typecheck  # tsc, no emit
```

There is no test runner — verification is `npm run build` (broken links throw) plus manual `npm run serve` to check light/dark mode and mobile widths.

## Layout

- Docusaurus **classic preset, TypeScript**, no blog. The site **root `/` is a custom React landing page** ([src/pages/index.tsx](src/pages/index.tsx) + `index.module.css`); guides are served under **`/docs`** (`routeBasePath: '/docs'`). [docs/index.mdx](docs/index.mdx) is the `/docs` "All guides" hub.
- Taxonomy groups are numbered folders under `docs/` (e.g. `03-system-cli/`), each with a `_category_.json` (label + sidebar position) and an `index.mdx` that acts as the category landing page (via Docusaurus's category-index convention) with an explicit `slug` (e.g. `/system-cli` → `/docs/system-cli`). The landing page and hub link to these slugs, so **keep the slug stable** if you rename a folder. The sidebar is **autogenerated** from this folder structure ([sidebars.ts](sidebars.ts)) — add a guide by dropping a `.mdx` file in the right folder; no sidebar edits needed.
- Full-text search is **self-hosted** via `@easyops-cn/docusaurus-search-local` (configured in `themes`, not the preset). No Algolia account.
- React-in-MDX components live in `src/components/` (e.g. `OSTabs`). Interactive command generators (charter milestone 3) go here too. Import in MDX with `@site/src/components/...`.
- `url`/`baseUrl` in [docusaurus.config.ts](docusaurus.config.ts) are placeholders pending the hosting decision (see Open decisions).

## Authoring conventions

- New guides start by copying [docs/\_template.mdx](docs/_template.mdx). Files/folders prefixed with `_` are **excluded from the build** by Docusaurus — that's why the template never renders. Don't rename a real guide to start with `_`.
- The template enforces the charter's guide skeleton (outcome title → summary → prerequisites → steps → verify → troubleshooting → optional generator → references).
- **Guides are cross-platform by default.** There is no per-guide platform badge. Write the shared narrative once; wrap *only* the OS-specific parts (install commands, file paths) in `<OSTabs>` with `<TabItem value="linux|macos|windows">`. `OSTabs` ([src/components/OSTabs](src/components/OSTabs/index.tsx)) pins `groupId="os"` so all switchers sync and persist — keep those three values consistent.
- Don't create a new taxonomy folder until ~2–3 guides need it (charter rule).
- Admonition titles must be **bracketed** — `:::info[Title]`, not `:::info Title`. The bracket-less form is invalid directive syntax and renders as literal `:::` text instead of a styled callout.

## What this project is

A personal, self-hosted knowledge base of software/tech tutorials, built on **Docusaurus (React + MDX)** and deployed as a **static site**. The differentiator over plain Markdown notes is structured navigation, full-text search, and interactive **command-generator** components that assemble exact commands (e.g. `chmod`, `yt-dlp`) instead of making readers hand-edit flags.

## Hard constraints (from the charter)

These are explicit non-goals — do not introduce them without the owner asking:
- **No backend/database** — static site only. No user accounts, comments, or CMS.
- No multi-author/editorial pipelines, no auto-generated API docs, no analytics-heavy tracking or SEO work.

## Content architecture

- **Groups are by purpose**, not by date or OS. Each group maps to a sidebar category (a folder under `docs/`). Defined groups: Media & Downloading, Emulation & Gaming, System & CLI Utilities, Networking, Windows-specific, Self-hosting & Servers, Misc / Snippets.
- **Resist creating a new category until ~2–3 guides need it.**
- Cross-cutting discovery uses Docusaurus's built-in **tagging** (e.g. `video`, `cli`, `troubleshooting`), plus a per-guide **platform tag** (Linux / Windows / Cross-platform).
- A `_template.mdx` enforces the guide skeleton. Every guide follows it: outcome-oriented title → one-line summary → platform & prerequisites → numbered copy-pasteable steps → "verify it worked" → troubleshooting → optional interactive generator → references.
- Interactive generators are React components embedded in MDX. They must update output live on input change, provide a working copy button, and remain legible in dark mode.

## Definition of Done (use as a review checklist)

A **guide** is done only when: it's in the right category with an outcome-oriented title and platform tag; **every command has been actually run and verified on the stated platform**; commands are copy-pasteable with no unexplained placeholders; it has a troubleshooting section; it renders correctly in light/dark mode at desktop and mobile widths; it has no broken links; and it's tagged.

The **MVP/site** is done when: `npm run build` completes with **zero broken-link warnings**; it deploys and loads; sidebar reflects the taxonomy; full-text search works across all guides; dark-mode toggle works; at least one guide per active category meets the per-guide DoD; at least one command-generator component is live; and a `_template.mdx` plus a short "how to add a guide" note exist.

The charter's section 8 lists testable acceptance criteria (AC-1…AC-6) — treat them as the acceptance gate before calling the MVP shippable. Broken-link warnings are a build failure here, not a warning to ignore.
